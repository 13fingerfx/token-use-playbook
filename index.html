A lot of waste was not from coding. It came from rereading too much context, swallowing verbose command output, and working with raw data that could have been compacted first.

Do not make Codex read raw data if a 50-line summary would answer the question.

Eight moves

The Token-Slimming Playbook

1

Compact raw data first

Summarize logs, JSON dumps, CSVs, repo scans, and command output before Codex reads them. Keep only relevant fields, samples, top errors, deduped lines, and compact JSON.

2

Limit command output

Use `head`, `tail`, filtered searches, and explicit script limits by default. For big outputs, write to a file and inspect selected ranges.

3

Create helper scripts

Turn repeated inspection into tools like `repo_summary.py`, `compact_logs.py`, `scan_errors.py`, and `list_recent_changes.py`.

4

Maintain a handoff file

Keep current goals, important files, tried commands, known errors, decisions, next steps, and files Codex should avoid rereading.

5

Say what not to read

Exclude `node_modules`, `.venv`, `dist`, `build`, generated files, archives, and any unrelated modules unless they are needed for the task.

6

Prefer targeted snippets

Ask for the OAuth callback function, the failing test diff, or the repo entry point map instead of broad whole-file or whole-repo reads.

7

Compact working context

Periodically ask Codex for a short handoff note with only the facts needed to continue. Remove dead ends and repeated detail.

8

Keep answers concise

Request the patch, the reason, and the test result. Skip repeated plans unless the plan actually changed.

Make it stick

Use Hooks for Rules Codex Must Not Forget

Markdown instructions are useful, but fragile. For noisy commands that should usually be compacted, a Codex hook can block raw output and return a useful summary before that output enters the working context.

Prefer project hooks first

Install command guards per repo before making them global. That keeps experiments from changing unrelated projects.

Keep instructions concise

Before adding lines to agent instructions, check the existing guidance. A small wording change is often better than another rule.

Separate global from local

Put common behavior in global instructions. Move git, Cloudflare, deployment, and project-specific details into focused runbooks.

Avoid temp-file detours

Do not default to “read the temp file for the rest.” Return counts, top errors, and next suggested filters first so Codex asks a narrower question.

Keep raw output opt-in

Save raw output for auditability, but require an explicit reason and a targeted range, pattern, or timestamp before Codex reads it.

Prompts That Reduce Token Burn

These are deliberately direct. They give Codex boundaries before it gets curious.

Compact First

Before reading raw data, create a compact working view with only relevant fields, top errors, deduplicated lines, counts, and samples. Read that summary first.

Read Boundaries

Do not scan node_modules, .venv, dist, build, generated files, archives, or unrelated modules. Inspect only files needed for this bug.

Handoff Note

Compact the current findings into a short handoff note. Include current goal, important files, tried commands, known errors, decisions, next steps, and what not to reread.

Concise Patch

Be concise. Show only the patch summary, the reason, and verification. Do not restate the whole plan unless it changed.

Instruction Hygiene

When updating agent instructions, first check relevant existing guidance. Prefer a small precise wording change over adding new lines. Move non-global guidance into task runbooks or project-local files.

Runbook Split

Keep global AGENTS guidance short. For task-specific workflows, load focused runbooks such as git.md, deploy.md, cloudflare.md, or project-local instructions only when relevant.

Hook Summary

If a hook truncates output, do not immediately read the temp file. First summarize counts, top errors, matching filenames, and suggested narrower filters. Read raw output only by explicit range, timestamp, or pattern.
